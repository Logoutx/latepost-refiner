# Workflow return handling (Claude Code fast path)

The `Workflow({ scriptPath, args })` dispatch returns a result object. Walk these fields, then go straight to **Step 5 (wrap-up)** in `SKILL.md`.

## ⛔ First check `auditUnavailable` — a failed run, not a successful one

If top-level **`auditUnavailable`** (`[{ path, label }]`) is **non-empty**, the deterministic audit **could not run** for those 成稿 even after one retry (e.g. the sandbox audit sub-agent kept returning unparseable output, or a sandbox error). **The run has FAILED.** The refined transcripts — and any 总结/时间线/逻辑稿 derived from them — are **unaudited, therefore unverified, therefore not deliverable as-is**. Do **not** write a success summary. Instead: (1) state plainly that the audit did not run and the deliverables are unaudited/invalid, naming each file; (2) the files are still on disk (work preserved) — run the audit **by hand** for each: `node "<this skill dir>/audit_refined.mjs" --source "<src>" --refined "<out>"`, read its JSON, and only then decide whether each 成稿 (and its derivatives) can be trusted. "Audit unavailable" is never "audit passed"; a gate that is silently skipped is not a gate.

## If the run partially fails — do NOT re-refine

The pipeline is built so a **cheap gate agent** can never cost the expensive work: refine runs even when scout fails (it reads the source directly), and completeness is judged by the **deterministic source-aware audit** (no separate haiku check agent) — its `ending_missing` gate is what marks a file `incomplete`. But if the **harness itself** drops a long run mid-way (网络卡死等), the refined transcripts — and any deliverables — are **already on disk** under `<output>/Transcripts/` (and `逻辑顺序/`, root). Recover cheaply, never re-refining:

- **Resume**: re-invoke `Workflow({ scriptPath, args, resumeFromRunId: '<the run id>' })` — completed refine agents return cached, only the failed cheap steps re-run.
- **Or finish by hand**: if the refined files exist, run **only** the missing pieces against the on-disk refine — the deterministic audit (`node "<this skill dir>/audit_refined.mjs" --source "<src>" --refined "<out>"`) for completeness, and one deliverable subagent each for any missing 访谈总结 / 时间线 / 逻辑顺序稿. This is the "我手动补完 check + 交付物、没重跑精校" flow — that's the right move, not a full re-run.

1. **`glossary`** — `Write` it in full to `<output dir>/校对表.md` to archive. It is **cumulative** (it already incorporates any `priorGlossaryText` you passed), so writing it back simply supersedes the old file — no manual merge. Before writing, glance over the typesetting: verification sources, same-referent reasons, and other full-sentence Chinese notes should use Arabic numerals, Pangu spacing, and full-width curly quotes. Subagents mostly get this right; fix any stragglers.
2. **`failed`** — these files weren't produced. Do them by hand per the manual pipeline (`references/manual-steps.md`, Steps 1–2).
3. **`incomplete`** — the deterministic audit's `ending_missing` gate fired: the source's last sentence never surfaced in the 成稿, so the output didn't reach the ending. Each item is `{ path, note }`. Dispatch a refine subagent to continue to the end.
4. **`unchecked`** — completeness could **not** be determined: no source-aware audit ran for the file (the audit capability was absent, or the sandbox audit sub-agent didn't return parseable JSON). This is not the same as `incomplete` — don't treat the file as passed *or* as truncated; self-check against the source ending (run `audit_refined.mjs` by hand, or eyeball with `tail`).
4a. **`audit` (per refined entry) + `auditFailed` (top-level, §2)** — the source-aware audit now runs **inside** the pipeline, per file, right after refine/stitch and before logic/总结/时间线 (no longer only a manual Step-5 pass). Each `refined[i].audit` is `{ status, hardFindings, softFindings, repaired, anchorsAdded, auditUnavailable }`. A **hard** finding (`content_gap` — a source stretch that never surfaced in the 成稿 with no fold trace, the silent-omission failure; or `quote_style` — ASCII 直引号紧贴中文 / 「」『』) triggers **one** automatic repair attempt (a refine sub-agent patches only the named spots) + **one** re-audit. If it passes, `repaired: true` and you're done. If it's **still hard**, the file is listed in top-level **`auditFailed`** (`[{ path, findings }]`) and a visible `> ⚠【内容缺口：源文件第 X-Y 行…】` marker is dropped into the 成稿. For each `auditFailed` file: (1) tell the user which line ranges are missing and why (often a content-policy pass on a sensitive segment); (2) offer to re-refine **just those source line ranges** — on a different provider if it looks policy-driven — and hand-merge; never re-refine the whole file. Note: 总结/时间线/逻辑稿 were generated after the gate, so they inherit the marked gap — regenerate them once the content is backfilled. `auditUnavailable: true` means the audit couldn't run (e.g. the sandbox sub-agent didn't return parseable JSON) — treat the file as unverified and run `audit_refined.mjs` by hand. Model-inserted `未精校段` markers (RULES 14 last-resort placeholders) surface as `model_marker` findings; the audit's line numbers are authoritative, not the marker's.
4b. **Source anchors** — after the audit (and any gap annotation), run `… --anchors` per file: each `##` section gets `<!-- 源 L…-L… · 时间-时间 -->` under its heading (invisible when rendered). Provenance for the desk: quote → source lines → recording timestamp. Only the 成稿 carries anchors — 总结/时间线/逻辑稿 cite 小标题, inheriting traceability through them. Sections with no matched evidence are skipped by design.
5. **`scoutSuspect` / `scoutFailed`** (non-empty) — scouting was unreliable for these files: `scoutSuspect` = returned garbled content (still broken after a retry, usually a network-corrupted stream); `scoutFailed` = returned **nothing at all** (stalled, e.g. on an oversized merged file). **Either way the outputs are fine** — refine reads the source directly and ran regardless — but those files' glossary entries are missing/untrustworthy. Tell the user and suggest re-running scouting on just those files once the network is stable (or after splitting an oversized file).
6. **`headingConflicts`** — these files already had sub-headings but ran under 'none'. Fold into the Step 5 questions (keep / redo), re-running that file alone if needed.
7. **`suspectedDuplicates`** — groups written differently but suspected of being the same referent that the script did not auto-merge (e.g. 王勇 / 汪勇). Already folded into `openQuestions`; ask the user in Step 5 whether they're the same.
8. **`networkUnverified`** (non-empty) — items the verify agent **skipped via the circuit breaker, never actually verified** (as opposed to "checked but not found"). In the Step 5 batch, offer a **re-verify option** (“网络恢复了吗？要不要把这 N 项补查一遍”). On confirmation, dispatch **one** sonnet subagent to verify just those via WebSearch — same discipline: only resolved with cited evidence, never fabricate, stop on consecutive errors — and append the conclusions to a 「## 补核结论」 section at the end of `<output dir>/校对表.md`. If a re-verify **corrects a name/term spelling already used in the outputs**, tell the user and propose one targeted replacement. Names follow the same strong-name guard: when a re-verify conclusion conflicts with an output's strong name, only report it, don't silently change it.
9. **`openQuestions`** — fold into the single Step 5 batch. Now also carries **疑似转录误写、未核实** items: entities the scout flagged as likely ASR mishears (`suspect_asr`) that verify couldn't resolve — the class where a consistently mis-transcribed name has no variants and would otherwise slip the verify filter. The glossary marks each with ⚠; surface them so the user confirms the correct 写法 (and if confirmed, propose a targeted replacement in any output that used the wrong spelling).
10. **`logic`** (the logical-order rewrites) — already written by the workflow to `<output>/逻辑顺序/`. If any item's `missingSections` is non-empty, the per-section coverage check suspects that section was dropped from the rewrite; spot-check it and re-run that file alone if needed.
11. **`summary` / `timeline`** — already written to disk by the workflow. **Before declaring success, run the derivative-attribution guard on each one** (the workflow's fs-less sandbox can't, so it's a wrap-up step, like the M8 cross-file check): `node "<this skill dir>/audit_refined.mjs" --derivative "<output>/…时间线.md" --kind timeline --corpus "<源1>,<源2>,…,<output>/Transcripts/<成稿1>.md,…"` (and again with `--kind summary` for the 访谈总结; corpus = **this run's source transcripts + refined 成稿**). This is the highest-risk newsroom failure: a figure attributed to the interview that the interviewee never actually said. A **`derivative_attribution` hard finding (exit 1)** means a 【访谈】-tagged measured figure (money/mass/distance/percent/duration) is absent from the interview — do **not** ship it: fix the label to 【公开·待记者核实】 if it's really public, or drop the figure. `reporterVerify` items are public·待核 figures for the reporter to confirm; `review` items are unlabeled/复核 figures to spot-check. (Bare integers and date/year anchors are never hard-failed — FP discipline.)
12. **`crossFileConflicts` (M8 跨文件互证) — universal (CLI/web) edition only; this Claude Code edition does NOT produce it.** The batch-level cross-file numeric-consistency check (same entity + unit stated with conflicting values across two files, where each file is internally faithful to its own source) needs to read every refined file back plus the 校对表 — which the Workflow sandbox cannot do (no fs; refined text never enters the orchestration layer), so it runs in the CLI/web layer instead. To get the equivalent by hand after a CC run, once all 成稿 are on disk call `checkCrossFileClaims` from `audit_refined.mjs`, or just eyeball shared figures (founding year, headcount, revenue) across the batch's transcripts.
